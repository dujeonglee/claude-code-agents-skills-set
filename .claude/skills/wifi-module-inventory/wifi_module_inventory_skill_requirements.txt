--------------------------------------------------------------------------------
1. PURPOSE
--------------------------------------------------------------------------------

Given a Linux kernel WiFi driver source directory, this skill produces a
Module Inventory -- a structured, human-readable catalog of every logical
module in the driver.

The Module Inventory is the primary and central deliverable. Everything else
(metrics, diagrams, scores) exists only to enrich and support the inventory.
If a piece of information does not help a reader understand a module better,
it is out of scope.

Target reader: A WiFi driver engineer who is new to the codebase and wants
to understand how it is organized, or a technical leader evaluating where
the architecture needs improvement.


--------------------------------------------------------------------------------
2. INPUT
--------------------------------------------------------------------------------

A single local filesystem path pointing to the root of the driver source tree.

  Example : /home/user/linux/drivers/net/wireless/ath/ath10k

Files the skill reads:
  *.c          Primary analysis target
  *.h          Interface and type definition analysis
  Makefile     Supplementary hint for intentional build-unit groupings

No build environment, binaries, or compilation database is required.


--------------------------------------------------------------------------------
3. OUTPUT: THE MODULE INVENTORY
--------------------------------------------------------------------------------

The Module Inventory is a Markdown document. It contains one entry per
identified logical module. Entries are ordered from most foundational
(depended on by many) to most concrete (depends on many).

Each module entry has exactly two layers: Overview and Technical Detail.
These are described in full in Section 5.

Skeleton of the output document:

  # Module Inventory: <driver name>
  > <driver path> | <file count> files | <total LOC> lines | <date>

  ---

  ## 1. <Module Name>
  ### Overview
  ...
  ### Technical Detail
  ...

  ---

  ## 2. <Module Name>
  ...

  (one section per module, no other top-level sections)


--------------------------------------------------------------------------------
4. MODULE IDENTIFICATION
--------------------------------------------------------------------------------

Before writing the inventory, the skill must identify the module boundaries.
This is an analysis step, not part of the output.

Rule 1 -- Single Responsibility First
  A module is a group of files that together implement exactly one named
  responsibility. The responsibility must be expressible in one short sentence
  beginning with a verb.
  Examples:
    "Manages the 802.11 connection state machine (auth, assoc, deauth, roam)"
    "Processes inbound and outbound data frames (Tx/Rx queues, sk_buff)"
    "Implements the cfg80211 interface between the driver and the kernel"

Rule 2 -- Cohesion as Evidence
  Files belong to the same module when they share primary data structures
  or form a recognizable function-name prefix cluster.
  Examples of shared-struct evidence:
    sta_info, ieee80211_sub_if_data  -->  MAC Management
    sk_buff, ieee80211_tx_data       -->  Data Path
    cfg80211_bss                     -->  Scan & Discovery

Rule 3 -- File Name Patterns as Starting Hints
  These are hints, not rules. A file can override its pattern hint if
  symbol evidence points elsewhere.

  Pattern keywords         Candidate module
  ----------------------   --------------------------------
  scan, bss, probe         Scan & Discovery
  auth, assoc, sme, mlme   MAC Management
  tx, rx, xmit, ampdu      Data Path Tx/Rx
  phy, rf, cal, antenna    PHY Layer
  fw, hif, cmd, event      Firmware Interface
  pm, ps, wow, sleep       Power Management
  reg, regulatory, dfs     Regulatory
  debug, trace, stat       Debug & Diagnostics
  util, common, helper     Utility / Common
  cfg80211, nl80211        Configuration Interface

Rule 4 -- Makefile as Tiebreaker
  When a file matches no clear pattern and symbol evidence is ambiguous,
  the obj-y grouping in the Makefile determines module membership.

Rule 5 -- Flag Ambiguous Files
  Files that genuinely span two responsibilities are not forced into one
  module. They are assigned to the dominant one and flagged as
  "mixed responsibility" in the Technical Detail section.


--------------------------------------------------------------------------------
5. MODULE ENTRY FORMAT
--------------------------------------------------------------------------------

Each module in the inventory is written in the following exact format.
The two-layer structure (Overview / Technical Detail) is mandatory.

════════════════════════════════════════════════════════════════════════════════

  ## <N>. <Module Name>

  ### Overview

  **Responsibility**
  One sentence starting with a verb that describes what this module does.
  This sentence is the single most important thing in the entire entry.
  It must be precise enough that a reader can determine whether a random
  .c file belongs here or not.

  **Role in the Driver**
  Two to four sentences. Explains where this module sits in the overall
  driver architecture: what calls into it, what it calls, and why it
  exists as a separate module rather than being merged into a neighbor.

  **Key Abstractions**
  A short list (3-6 items) of the most important structs, enums, or
  callback tables that define this module's domain.
  Example:
    - ieee80211_sub_if_data  : per-vif state container
    - sta_info               : per-peer station record
    - ieee80211_ops          : hardware callback vtable

  ---

  ### Technical Detail

  **Files**
  A table with one row per file. Columns: File, Type, Responsibility.
  Type is one of: core | interface | helper | internal-header | public-header

  | File              | Type             | Responsibility                          |
  |-------------------|------------------|-----------------------------------------|
  | mlme.c            | core             | Main MLME state machine implementation  |
  | auth.c            | core             | Authentication frame handling           |
  | ieee80211_i.h     | internal-header  | Private structs shared within module    |
  | mac80211.h        | public-header    | API exposed to rest of driver           |

  **Internal Cohesion**
  One paragraph. Describes how tightly the files in this module are
  coupled to each other. Answers: Do they share a central data structure?
  Do they form a clear pipeline or state machine together? Are there any
  files that feel loosely attached?
  
  Cohesion rating (choose one and justify in the paragraph):
    STRONG   -- files form a clear unit; removing any one breaks the others
    MODERATE -- most files are tightly related; 1-2 are peripheral
    WEAK     -- files are grouped by topic but operate largely independently

  **External Dependencies**
  A table listing every other module this module depends on, with a brief
  description of what specifically is used from that module.

  | Depends On            | What Is Used                                      |
  |-----------------------|---------------------------------------------------|
  | Firmware Interface    | ath10k_wmi_cmd() to send auth/assoc commands      |
  | Utility / Common      | ath10k_dbg() logging, memory allocation helpers   |

  If this module has no external dependencies: write "None (self-contained)."

  **Issues** (omit this section entirely if there are no issues)
  A bulleted list of architectural problems observed in this module.
  Each item names the problem, the affected files, and the consequence.
  Use one of these labels:
    [SRP VIOLATION]    -- module is doing more than one thing
    [WEAK COHESION]    -- files don't naturally belong together
    [HIGH COUPLING]    -- too many external dependencies
    [CIRCULAR DEP]     -- this module and another depend on each other
    [INFORMATION LEAK] -- internal structs exposed in public headers

  Example:
    - [SRP VIOLATION] rate_control.c handles both rate selection and
      TX retry logic. These are distinct responsibilities and should be
      separated. Risk: changes to retry policy require touching rate
      selection code.

════════════════════════════════════════════════════════════════════════════════


--------------------------------------------------------------------------------
6. COHESION RATING CRITERIA
--------------------------------------------------------------------------------

STRONG cohesion -- all three conditions are true:
  a) Files share at least one central data structure as their primary state
  b) Functions in one file are directly called by other files in the module
  c) The module has a clearly identifiable entry point or public API surface

MODERATE cohesion -- condition (a) is true but (b) or (c) is partial:
  Files access the same structs but mostly operate in parallel rather than
  forming a pipeline. Typical of "management" modules.

WEAK cohesion -- condition (a) is absent:
  Files are grouped because they deal with the same topic, but each could
  be moved to another module with minimal impact. Usually a sign that the
  module boundary should be reconsidered.


--------------------------------------------------------------------------------
7. EXTERNAL DEPENDENCY RULES
--------------------------------------------------------------------------------

What counts as an external dependency:
  - A #include of a header owned by a different module
  - A direct function call to a function defined in a different module
  - Access to a global variable or struct defined in a different module

What does NOT count:
  - Includes of kernel headers (<linux/*, <net/*)
  - Includes of the driver's own utility/common module (listed separately)
  - Indirect dependencies (A depends on B which depends on C; C is not
    listed as a dependency of A)

Dependency direction convention:
  "Module A depends on Module B" means A uses something from B.
  This is written as A --> B in any diagram context.


--------------------------------------------------------------------------------
8. ISSUE LABELS -- DETECTION CRITERIA
--------------------------------------------------------------------------------

[SRP VIOLATION]
  Triggered when a single file contains functions that belong to two
  distinct responsibility domains as defined in Section 4, Rule 1.
  Or when the module's own responsibility sentence requires "and" to
  describe what it does.

[WEAK COHESION]
  Triggered when the cohesion rating is WEAK (see Section 6).
  Also triggered when more than 30% of files in the module have no
  direct symbol relationship to any other file in the module.

[HIGH COUPLING]
  Triggered when the External Dependencies table has 5 or more rows.
  (Utility/Common excluded from the count.)

[CIRCULAR DEP]
  Triggered when Module A lists Module B as a dependency AND
  Module B lists Module A as a dependency (direct cycle).
  Or when a cycle of length 3+ is detected (A->B->C->A).
  Detection method: Tarjan's Strongly Connected Components algorithm
  applied to the inter-module dependency graph.

[INFORMATION LEAK]
  Triggered when a public-header file exposes structs or typedefs that
  are only meaningful within the module's internal implementation
  (i.e., no external module needs to allocate or inspect the struct
  directly, yet it is exposed in a public header).


--------------------------------------------------------------------------------
9. WHAT IS EXPLICITLY OUT OF SCOPE
--------------------------------------------------------------------------------

The following are intentionally excluded from this skill. They were
considered and rejected to keep the Module Inventory focused and readable.

  - SAM score, MCD score, or any aggregate numeric score
    Reason: Scores compress information that is more useful when shown
            per-module in plain language.

  - Mermaid or other dependency diagrams
    Reason: The External Dependencies tables in each module entry convey
            the same information in a more precise, readable form.
            A diagram can be generated separately if needed.

  - Improvement roadmap or prioritized refactoring backlog
    Reason: The Issues section in each module entry is the correct place
            for actionable findings. A separate roadmap section duplicates
            this and dilutes focus.

  - Inter-module dependency matrix (N x N table)
    Reason: Redundant with the per-module External Dependencies tables.

  - Lines of code, file count, or other size metrics
    Reason: Size is informational but does not help a reader understand
            what a module does or whether it is well-designed.
            Exception: total driver LOC is shown in the document header
            for orientation purposes only.


--------------------------------------------------------------------------------
10. EXAMPLE MODULE ENTRY (for reference during implementation)
--------------------------------------------------------------------------------

The following is a realistic example entry for a mac80211-based driver.
This is not prescriptive -- the actual content will vary by driver.

────────────────────────────────────────────────────────────────────────────────

## 3. MAC Management

### Overview

**Responsibility**
Manages the 802.11 connection state machine, including authentication,
association, roaming, and disconnection for all virtual interfaces.

**Role in the Driver**
MAC Management is the behavioral core of the driver. The Configuration
Interface module translates cfg80211 commands into requests that land here.
This module drives the connection lifecycle by sending commands to the
Firmware Interface module and reacting to events from it. No other module
is allowed to initiate a connection or authentication sequence -- all such
logic is owned here.

**Key Abstractions**
  - ieee80211_sub_if_data  : per-vif state container (auth state, assoc state)
  - sta_info               : per-peer station record and capability cache
  - ieee80211_mgd_data     : managed-mode specific state (timers, retry counts)
  - cfg80211_connect_params: connect request parameters from userspace

---

### Technical Detail

**Files**

| File              | Type            | Responsibility                                  |
|-------------------|-----------------|-------------------------------------------------|
| mlme.c            | core            | Main MLME state machine, event dispatch         |
| auth.c            | core            | Authentication frame construction and parsing   |
| assoc.c           | core            | Association request/response handling           |
| sme.c             | core            | SME layer: translates cfg80211 connect calls    |
| ibss.c            | core            | IBSS (ad-hoc) mode connection management        |
| ieee80211_i.h     | internal-header | Private structs shared across module files      |

**Internal Cohesion**
STRONG. All five .c files operate on ieee80211_sub_if_data as their primary
state container, and form a clear sequential pipeline: sme.c receives the
connect request, auth.c handles the authentication exchange, assoc.c
completes association, and mlme.c owns the state machine that coordinates
all of them. ibss.c is slightly peripheral (ad-hoc only) but still operates
on the same central struct.

**External Dependencies**

| Depends On            | What Is Used                                         |
|-----------------------|------------------------------------------------------|
| Firmware Interface    | wmi_send_cmd() to transmit auth/assoc frames via FW  |
| Configuration Iface   | cfg80211_connect_result() to report outcome upstream |
| Utility / Common      | Logging macros, timer helpers, memory allocation     |

**Issues**

  - [SRP VIOLATION] ibss.c mixes ad-hoc beacon generation with ad-hoc
    connection management. Beacon generation is a PHY/MAC-boundary concern
    and would more naturally live adjacent to the AP-mode beacon logic.
    Current placement makes it harder to disable IBSS support at compile time.

  - [INFORMATION LEAK] ieee80211_i.h exposes ieee80211_mgd_data in full
    to all files that include it, including files in Data Path and Scan.
    Only MAC Management files should need the internals of this struct.

────────────────────────────────────────────────────────────────────────────────


--------------------------------------------------------------------------------
11. ACCEPTANCE CRITERIA
--------------------------------------------------------------------------------

AC-1  Every identified module has both Overview and Technical Detail sections
      with no empty or placeholder fields.

AC-2  Every module's Responsibility statement is a single sentence starting
      with a verb, specific enough to serve as a membership test for files.

AC-3  Every file in the source directory is assigned to exactly one module.
      No file is omitted.

AC-4  Every entry in an External Dependencies table names a specific function,
      struct, or header -- not just a module name.

AC-5  Cohesion ratings are consistent: a module rated STRONG must satisfy
      all three conditions in Section 6. The reasoning is written in the
      Internal Cohesion paragraph.

AC-6  Issues section is omitted entirely when there are no issues.
      It is never present but empty.

AC-7  On a reference driver (ath10k), a domain expert reviewing the inventory
      agrees with at least 80% of module assignments and responsibility
      statements without modification.
